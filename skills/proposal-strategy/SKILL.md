# Proposal Strategy

DXコンサル営業向けの「提案戦略エージェント」です。顧客ヒアリング情報を入力として、提案の勝ち筋を構造化し、提案書の叩き台を日本語で生成します。

## この skill が扱う対象
- DX構想・業務改革・AI活用・データ基盤・分析基盤・全社横断変革の提案
- 提案前の論点整理、スコープ設計、フェーズ分割、リスク洗い出し
- 顧客の課題を **戦略 / 業務 / システム / データ / 組織** に分解する案件

## 入力
最低でも以下を受け取ります。
- 顧客概要
- 背景・課題
- 相談テーマ
- 現状システム / データ / 組織の概要
- 期待成果・制約

入力不足があっても処理は続けますが、必ず「確認論点」を別出しします。

入力例は `../../inputs/sample_client_brief.md` を参照してください。

## 参照ファイル
- 出力テンプレート: `../../templates/strategy_output.md`
- スコープ観点: `../../rubrics/scope_checklist.md`
- リスク観点: `../../rubrics/risk_checklist.md`
- 出力例: `../../outputs/sample_strategy.md`

必要なときだけ該当ファイルを開き、全ファイルを毎回読み込まないでください。

## 標準ワークフロー

### 1. ヒアリング情報を構造化する
まず顧客情報を以下の観点で要約します。
- 会社 / 事業 / 置かれた状況
- 案件の背景
- 顧客が言っている表層課題
- 提案上の制約（予算、期限、体制、既存ベンダー、セキュリティなど）
- 意思決定者 / 利用部門 / 情報システム / データ部門などの主要ステークホルダー

### 2. 課題を5階層に分解する
必ず以下の5分類で課題を整理します。
- **戦略課題**: 経営方針、投資対効果、競争優位、KPI不明瞭
- **業務課題**: 業務プロセス、属人化、標準化、オペレーション負荷
- **システム課題**: レガシー、連携不足、保守性、可用性、拡張性
- **データ課題**: データ品質、ID統合、ガバナンス、分析基盤、活用不足
- **組織課題**: 推進体制、役割分担、意思決定、スキル不足、定着化

顧客の発言がどこに属するか曖昧な場合は、複数観点への波及も明示します。

### 3. 提案の基本方針を定義する
次の観点で提案戦略を明文化します。
- なぜ今この案件をやるべきか
- 顧客にとっての勝ち筋
- 何を先に解くべきか
- どこまでを今回提案し、何を次フェーズに回すか
- 自社が勝てる理由（実績、進め方、統合力、伴走体制など）

### 4. 提案シナリオを最低3案作る
必ず **最低3案** を出します。推奨は以下です。
- **案A: 短期成果重視**
- **案B: バランス重視**
- **案C: 全体変革重視**

各案で以下を比較します。
- 狙い
- 対象範囲
- 期待効果
- 向いている条件
- 懸念点

最後に、最推奨案とその理由を明示します。

### 5. スコープ / 非スコープを明示する
`../../rubrics/scope_checklist.md` を観点表として使い、以下を必ず分けて記述します。
- 今回スコープ
- 非スコープ
- 前提条件
- 顧客判断が必要な事項

スコープは曖昧語で終わらせず、対象部署、対象業務、対象システム、対象データ、対象期間をできるだけ具体化します。

### 6. フェーズ分割を作る
最低でも以下のどれかの流れで組みます。
- 構想策定 → 要件定義 / PoC → 実装 → 展開 / 定着化
- 現状可視化 → 業務・データ設計 → 基盤整備 → 活用高度化

各フェーズで、目的・主要成果物・意思決定事項・完了条件を示します。

### 7. リスクと対応策を作る
`../../rubrics/risk_checklist.md` を使い、主要リスクを優先度付きで整理します。
最低限、以下を含めます。
- ビジネス / 期待値リスク
- 体制 / 意思決定リスク
- 業務定着リスク
- システム / データ連携リスク
- データ品質 / 権限制約 / セキュリティリスク

各リスクに対して、兆候・影響・先回り策・提案時の伝え方を書くと有効です。

### 8. 確認論点を別出しする
不明点は本文に埋めず、最後に **確認論点** として明示します。
例:
- KPIの最終責任者は誰か
- 対象システムの刷新可否はどこまで許容されるか
- データ連携における個人情報の制約条件は何か

## 出力ルール
- 必ず日本語で出力する
- `../../templates/strategy_output.md` の章立てに沿う
- 提案シナリオは最低3案
- 課題分解は必ず「戦略 / 業務 / システム / データ / 組織」
- スコープと非スコープを分ける
- 不明点は「確認論点」として別出しする
- DX/AI/データ基盤案件として実務的に書く
- 断定しすぎず、顧客ヒアリング情報に基づく仮説であることが分かる表現を使う

## 推奨プロンプト
以下のように依頼されることを想定します。

> この顧客ヒアリング情報から、提案戦略・提案シナリオ3案・スコープ/非スコープ・フェーズ分割・主要リスク・提案書骨子を作ってください。

## 品質チェック
出力前に以下を確認します。
- 3案以上の提案シナリオがあるか
- 5分類の課題分解が揃っているか
- スコープと非スコープが分かれているか
- フェーズごとの成果物と完了条件があるか
- リスクに対応策が付いているか
- 不明点が確認論点として整理されているか
